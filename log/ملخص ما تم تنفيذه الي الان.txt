## 🚀 الخطوة التالية (Next Step):

المطلوب الآن بناء **Orders API** (نظام الطلبات).

### المطلوب في Orders API:

**الـ Endpoints المخطط لها:**
| Method | Endpoint | الوصف |
|--------|----------|-------|
| GET | /api/orders | عرض كل طلبات المستخدم (مع pagination) |
| GET | /api/orders/{id} | عرض طلب معين بالتفصيل |
| POST | /api/orders | إنشاء طلب جديد من السلة |
| PUT | /api/orders/{id}/status | تحديث حالة الطلب (Admin only) |
| DELETE | /api/orders/{id} | إلغاء طلب (في حالات محددة) |
| GET | /api/orders/admin/all | عرض كل الطلبات (Admin only) |

**القواعد المطلوب مراعاتها:**
1. عند إنشاء طلب:
   - يتم نقل المنتجات من `cart_items` إلى `order_items`
   - يتم تخزين نسخة (snapshot) من بيانات المنتج (السعر، الاسم) لحماية الطلب من التغيير لاحقاً
   - يتم خصم الكمية من المخزون (stock_quantity)
   - يتم تسجيل الحركة في `inventory_logs`
   - يتم تفريغ السلة بعد إنشاء الطلب

2. حالات الطلب (order status):
   - pending (قيد الانتظار)
   - processing (قيد المعالجة)
   - completed (مكتمل)
   - cancelled (ملغي)
   - refunded (مسترجع)

3. لا يمكن تعديل طلب بعد إنشائه (للحفاظ على integrity)
4. لا يمكن إلغاء طلب إذا كان completed أو cancelled

**الجداول المعنية:**
- `orders` (الطلبات)
- `order_items` (منتجات الطلب مع الـ snapshot)
- `carts` و `cart_items` (لتفريغها بعد إنشاء الطلب)
- `inventory_logs` (لتسجيل حركة المخزون)
- `products` (لتحديث المخزون)

## 📌 أسلوب التطبيق المطلوب:

- نفس **Service-Repository Pattern** المستخدم في Products و Cart
- استخدام **Laravel Sanctum** للمصادقة
- الـ Admin فقط يمكنه تحديث حالة الطلب
- الـ API Responses تكون بصيغة JSON موحدة:
```json
{
    "success": true,
    "data": {},
    "message": "",
    "errors": {}
}
⚠️ ملاحظات مهمة:
لا تستخدم SoftDeletes (لأن قاعدة البيانات لا تحتوي على deleted_at)

لا تغير هيكل قاعدة البيانات بدون الرجوع إليّ

اتبع العلاقات المحددة في الـ Schema الأصلي للمشروع

دعم الأداء لـ 100,000+ منتج (استخدام Eager Loading، Indexes)

استخدام try-catch في الـ Controller للتعامل مع الـ Exceptions
