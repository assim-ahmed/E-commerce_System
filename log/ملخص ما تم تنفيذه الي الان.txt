📋 القواعد المطلوب مراعاتها:
شروط إضافة التقييم:

المستخدم يجب أن يكون مسجل دخول (Authenticated via Sanctum)

المستخدم يجب أن يكون قد اشترى المنتج فعلاً (يوجد order كامل بهذا المنتج)

المستخدم لا يمكنه تقييم نفس المنتج أكثر من مرة

ربط التقييم بـ order_id (للتأكد من الشراء)

بيانات التقييم:

rating من 1 إلى 5 نجوم (required, integer, min:1, max:5)

comment نص (nullable, string, max:1000)

images صور (nullable, array, max:5, each image: jpg,jpeg,png,max:2MB)

is_approved افتراضياً false، فقط Admin يوافق عليه

عرض التقييمات:

المستخدم العادي يرى فقط التقييمات المعتمدة (is_approved = true)

Admin يرى كل التقييمات (حتى غير المعتمدة)

صاحب التقييم يرى تقييمه حتى لو غير معتمد

التعديل والحذف:

يمكن للمستخدم تعديل أو حذف تقييمه فقط

لا يمكن التعديل بعد موافقة Admin (يُرجع 403)

عند حذف المنتج، يتم حذف تقييماته (Cascade)

الإحصائيات:

كل منتج يحتوي على average_rating (يتم حسابه عند إضافة/تعديل/حذف/موافقة تقييم)

reviews_count عدد التقييمات المعتمدة فقط

🗃️ الجداول المعنية:
reviews (جدول التقييمات الرئيسي)

user_id, product_id, order_id, rating, comment, images, is_approved

products (لتحديث average_rating و reviews_count)

orders (للتحقق من شراء المستخدم للمنتج - status = completed)

order_items (للتأكد أن المنتج موجود في طلب مكتمل)

🔄 سير العمل (Workflow):
text
1. مستخدم يشتري منتج (Order status = completed)
                ↓
2. المستخدم يذهب إلى المنتج ويضيف تقييم
   POST /api/products/{id}/reviews
                ↓
3. النظام يتحقق:
   - هل المستخدم اشترى هذا المنتج؟
   - هل سبق له التقييم؟
                ↓
4. التقييم يُخزن في DB مع is_approved = false
                ↓
5. Admin يرى التقييم في لوحة التحكم
   PUT /api/admin/reviews/{id}/approve
                ↓
6. بعد الموافقة، التقييم يظهر للجميع
   ويتم تحديث average_rating للمنتج
📌 أسلوب التطبيق المطلوب:
نفس Service-Repository Pattern المستخدم في Products و Cart و Orders

استخدام Laravel Sanctum للمصادقة

الـ Admin فقط يمكنه الموافقة على التقييمات

الـ API Responses تكون بصيغة JSON موحدة:

json
{
    "success": true,
    "data": {},
    "message": "",
    "errors": {}
}
⚠️ ملاحظات مهمة:
لا تستخدم SoftDeletes (لأن قاعدة البيانات لا تحتوي على deleted_at)

لا تغير هيكل قاعدة البيانات بدون الرجوع إليّ

اتبع العلاقات المحددة في الـ Schema الأصلي للمشروع

دعم الأداء لـ 100,000+ منتج (استخدام Eager Loading، Indexes)

استخدام try-catch في الـ Controller للتعامل مع الـ Exceptions

عند الموافقة على تقييم (is_approved = true)، يتم إعادة حساب average_rating للمنتج

🎯 المطلوب منك الآن:
بناء Reviews API كاملاً بنفس جودة وطريقة Cart API و Orders API، مع الالتزام بكل القواعد المذكورة أعلاه.
